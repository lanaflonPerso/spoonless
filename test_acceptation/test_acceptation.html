Les tests d’acception¶

Quelle que soit la méthode de développement retenue pour développer une solution logicielle, il est toujours nécessaire d’effectuer la qualification du produit. La qualification correspond à toutes activités réalisées pour vérifier qu’un système logiciel est conforme aux attentes.

Les tests d’acceptation s’inscrivent dans une démarche de qualification pour déterminer si le produit est conforme dans son utilisation aux objectifs fixés. Il s’agit d’une description formelle du comportement d’un logiciel sous la forme d’un scénario.

Recette fonctionnelle¶

Dans l’approche de gestion de projet, la recette fonctionnelle (ou plus simplement la recette) est une des phases du développement logiciel. Son objectif est d’attester que le contrat entre MOA/MOE a bien été respecté par la réalisation de sessions de test.

La recette fonctionnelle est un moyen de vérification de la conformité d’un SI livré par rapport aux spécifications émises par la MOA. Les tests d’acceptation (souvent appelés tests fonctionnels) servent à réaliser cette vérification.

Test d’acceptation agile¶

Dans une approche agile, le produit est le plus souvent découpé en user stories. Chaque user story doit être testable, c’est-à-dire qu’elle définit ses critères d’acceptation. Un test d’acceptation aide donc à la fois, à comprendre la finalité d’une user story mais également à vérifier, après son implémentation, que la réalisation est conforme à ce qui a été défini.

Dans une approche agile, l’acceptation d’une user story fait partie intégrante de son cycle de vie est intervient dans la même itération que son développement.

Note

L’eXtreme Programming propose une approche ATTD (Acceptance Test Driven Development). Il s’agit de définit les tests d’acceptation avant le développement d’une user story de manière à guider les développements au plus près des attentes des utilisateurs.

Gojko Adzic dans son ouvrage Specification by Examples propose de revoir la façon de spécifier un logiciel en utilisant pour cela une série d’exemples qui correspondent de fait à des tests d’acceptation.

Notion de cas de test¶

Un cas de test désigne une interaction avec le système destinée à produire un résultat attendu.

Un cas de test est généralement décrit par son titre. Il contient une liste d’étapes à réaliser. Chaque étape doit correspondre à une action de l’utilisateur et s’accompagne d’un résultat attendu (et si possible un seul). Le résultat attendu doit être explicité clairement et ne doit pas être interprétable.

Il faut limiter le nombre d’étapes d’un cas de test pour en réduire sa complexité. Entre chaque étape, il ne peut pas y avoir de choix ou d’alternative laisser au testeur (faire ceci ou cela). La succession des étapes doit être non ambiguë.

Cas de test : afficher son profil utilisateur

Étape : Sélectionner le menu utilisateur

Résultat attendu : Le menu s’affiche et contient l’entrée « Profil »

Étape : Sélectionner l’entrée « Profil »

Résultat attendu : Le détail du profil s’affiche dans une popup avec le nom, le prénom et la photo, les boutons « modifier » et « fermer » sont visibles et cliquables.

Cas de test : éditer le profil utilisateur

Étape : Cliquer sur le bouton « modifier »

Résultat attendu : Les champs nom et prénom deviennent éditables, le bouton « modifier » disparaît et le bouton « sauvegarder » apparaît.

Étape : Modifier le nom du profil

Résultat attendu : Le champ est mis à jour

Étape : Cliquer sur le bouton « fermer »

Résultat attendu : La popup du profil utilisateur se ferme.

Il ne faut pas croire qu’il existerait objectivement un niveau idéal de rédaction pour les cas d’utilisation. Il est nécessaire d’avoir une idée du profil des personnes qui seront chargées d’effectuer les tests et de valider chaque étape. Le niveau de rédaction nécessaire est donc largement dépendant de l’équipe qui participera aux sessions de test. S’il s’agit d’une équipe spécialisée pour la réalisation des tests mais n’ayant que très peu d’implication dans le développement du système, alors les cas de tests devront être extrêmement détaillés pour guider au mieux les testeurs. A contrario, si les personnes réalisant les tests travaillent quotidiennement avec les développeurs et/ou les experts du domaine, les cas de test pourront contenir des termes propres au domaine de l’application et synthétiser plus facilement chaque étape en profitant d’un savoir commun de l’ensemble de l’équipe. De même, pour les personnes en charge de participer aux sessions de test, il faut tenir compte du savoir-faire technique dans la rédaction des cas de tests: l’ambiguïté d’un cas de test peut reposer sur une asymétrie des savoirs techniques entre le rédacteur et le testeur. Par exemple, si une étape d’un cas de test nécessite de vérifier le code retour d’un programme, le testeur doit avoir une idée de ce qu’est un code retour et comment l’obtenir.

Scénario de test¶

Un scénario de test est le regroupement de cas de test. Alors qu’un cas de test représente une interaction limitée avec le système, un scénario représente la réalisation d’une fonctionnalité complète.

Un ensemble de scénario permet de s’assurer qu’une fonctionnalité fournie est conforme aux exigences (méthode de gestion de projet) ou à la user story (méthode agile).
